# Диагностика (полная)

## Результаты проверок

| Проверка | Результат |
|----------|-----------|
| Backend GET /mini-app/init | **OK** — 405 Method Not Allowed (ожидаемо, нужен POST). |
| Backend POST /mini-app/init (пустой initData) | **OK** — 401 `{"reason":"initData пустой"}`. |
| CORS OPTIONS /mini-app/save-test-result (Origin: Vercel) | **OK** — 204, `Access-Control-Allow-Origin: *`, `Access-Control-Allow-Methods: GET, POST, OPTIONS`. |
| Vercel Production URL | **ПРОБЛЕМА** — 404, `x-vercel-error: DEPLOYMENT_NOT_FOUND`. Нет успешного деплоя или production-алиас не привязан. |
| POST save-test-result (фейк token) | **OK** — 401 `{"error":"Invalid token"}`. |
| POST test-history (фейк token) | **OK** — 401 `{"error":"Invalid token"}`. |

## Вывод

**Проблемы (с причинами):**
1. **Vercel 404 (DEPLOYMENT_NOT_FOUND)** — по Production URL не привязан ни один деплой. Возможные причины: сборка после push не запускалась или падала; production branch не указан; деплой не помечен как Production. **Исправление:** успешный деплой в Vercel и привязка к домену (Redeploy в Dashboard или через API с VERCEL_TOKEN).
2. **В Telegram кнопка не открывает app** — из-за п.1 (URL отдаёт 404).
3. **Полный E2E с реальным токеном не проверен** — токен выдаётся только из POST /mini-app/init при валидном Telegram initData (подпись BOT_TOKEN). Без реального открытия Mini App из бота симулировать нельзя. После появления рабочего URL цепочка (init → token → save → history) должна работать по контракту.

**Бэкенд и CORS в порядке. HTTP (не HTTPS) не блокирует работу; при необходимости позже можно поднять HTTPS на сервере и обновить VITE_BOT_BACKEND_URL.**

Готов исправлять. Для автоматического исправления Vercel через API нужен **VERCEL_TOKEN** — пришли токен, либо сделай вручную Redeploy в Vercel Dashboard.
