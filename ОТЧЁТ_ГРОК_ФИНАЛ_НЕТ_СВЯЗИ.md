## Шаг 1. Диагностика сети и связи

### Выполнено

1. **curl app URL**  
   `https://ozharov164-glitch.github.io/cozyreset-path-to-self-miniapp/` → **HTTP 200**, ~0.4 c.

2. **curl backend (Tunnel)**  
   `POST https://bath-instructional-enclosure-cite.trycloudflare.com/mini-app/init` с телом `{"initData":"fake"}` → **HTTP 401** за ~0.44 c (ожидаемо: неверная подпись). Таймаутов нет, туннель доступен с проверяющей сети.

3. **Туннель на сервере**  
   SSH 217.114.11.97:  
   - `systemctl status cloudflared-tunnel-bot` → **active (running)**.  
   - URL из journalctl: `https://bath-instructional-enclosure-cite.trycloudflare.com`.  
   - В логах туннеля были старые ошибки «connection refused» к 127.0.0.1:8080 (в момент перезапуска бота).

4. **Гео**  
   Симуляция запросов с РФ-IP не выполнялась. Возможный геоблок Cloudflare в РФ учтён в рекомендациях (домен + nginx).

5. **Симуляция WebApp**  
   Полноценный E2E с валидным initData не выполнен (нужен реальный initData от Telegram). Проверены только доступность app (200) и ответ бэкенда на init/history (401 без токена).

6. **Debug в приложении**  
   В `path-to-self-miniapp/src/api/client.ts` добавлены логи под префиксом `[PTS]`:  
   - при загрузке конфига: `location.href`, `location.search`, `backend` из query, итоговый `backendUrlOverride`;  
   - в ensureAuth: длина `initData`, `getBackendUrl()`, номер попытки, ошибки fetch.  
   Коммит запушен в репо приложения (GitHub Pages деплой по push в main).

**Диагноз шага 1:** Туннель жив, backend отвечает. Возможные причины «Нет связи» у пользователя: неверный/пустой backend URL в приложении (query обрезан Telegram?), таймаут из РФ (гео), временная недоступность бота (connection refused в логах туннеля). Готовы фиксы по шагам 2–3.

## Шаг 2. Фикс Tunnel/HTTPS (стабильный URL)

### Выполнено

- Текущий URL туннеля совпадает с тем, что в `config.json`: `https://bath-instructional-enclosure-cite.trycloudflare.com`. Дополнительно бот передаёт этот URL в ссылке кнопки «Путь к Себе» как `?backend=...` (уже было реализовано ранее).
- Let's Encrypt без домена (только по IP) невозможен. Для обхода геоблока/нестабильности Cloudflare зафиксирована инструкция: завести домен (например DuckDNS), выполнить `DOMAIN=... ./scripts/setup-https-nginx.sh` в репо бота, затем обновить в приложении `public/config.json` и на сервере `MINI_APP_BACKEND_HTTPS_URL` на `https://<домен>`.

**Фикс шага 2:** Туннель оставлен как основной вариант. config.json актуален. Для РФ при необходимости — переход на nginx + домен по инструкции выше.

## Шаг 3. Фикс связи и auth (устойчивость)

### Выполнено

1. **client.ts — ensureAuth**  
   - До 5 попыток POST `/mini-app/init` с задержками 0, 100, 300, 1000, 2500 мс.  
   - При 401 или 5xx не выходим сразу, переходим к следующей попытке.  
   - Токен по возможности сохраняется в store.

2. **client.ts — apiSaveTestResult**  
   - При ответе с `error: 'network'`: сброс токена в store, пауза 300 мс, повторный вызов `ensureAuth()`, затем повтор одной попытки save.  
   - В теле запроса по-прежнему уходят и token, и initData (если есть).

3. **Бэкенд (yookassa_service.py)**  
   - В обработчике save-test-result: лог входа с указанием наличия token и длины initData.  
   - При 401 (user_id не определён): запись в `logs/mini_app_errors.log` через `_append_mini_app_error("save-test-result", "401 no user", ...)`.  
   - При ошибке разбора initData — туда же.

4. **Коммит и деплой приложения**  
   Коммит: «fix: enhance connection stability — 5 retries init with backoff, save retry after re-auth». Push в main репо приложения (деплой на GitHub Pages автоматически).

**Фикс шага 3:** Связь и auth усилены: несколько попыток init, повтор save после re-auth, доп. логи на бэкенде для разбора 401 и сетевых сбоев.
